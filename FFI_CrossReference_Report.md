## 5. 经验教训：类型重复定义风险

从长期记忆中检索到一个重要教训（记录于 2026-06-09）：

> 在同一项目中存在两个同名的 `RoomNotificationMode` enum：一个在 `Generated/matrix_sdk_ffi.swift`（FFI 生成，无 rawValue），另一个在 `SocialApp/Services/Spaces/RoomListService.swift`（本地定义，带 String rawValue 和 CaseIterable 遵循）。若两者处于同一编译模块，将导致 `Invalid redeclaration` 编译错误。

**建议**：验证 FFI 参数类型时，不能只看方法签名，必须同时检查参数类型本身是否存在本地重复定义。

**建议：是，建议重新生成。**

理由：
1. `stub.c` 已经更新（今天），说明 Rust 侧有变更
2. Swift 文件与 C 桥接层不同步可能导致运行时崩溃或链接错误

### 6.3 存在的具体问题

1. **时间不同步**：stub.c (2026-06-13) vs Swift (2026-06-08)，相差 5 天
2. **潜在编译/运行时风险**：如果 Rust 侧新增/修改了方法签名，旧的 Swift 绑定可能：
   - 缺少新方法 → 编译时 "method not found" 错误
