## 5. 为什么文件数量多？

### 5.1 根本原因：UniFFI 的多 Crate 独立导出策略

Matrix Rust SDK 由 7 个 Rust crate 组成，其中 6 个通过 UniFFI 导出到 iOS：

```
matrix-sdk (核心 Rust crate)
  ├── matrix-sdk-base
  ├── matrix-sdk-common
  ├── matrix-sdk-crypto
  └── matrix-sdk-ui

matrix-sdk-ffi (FFI 桥接 crate，聚合以上所有)
```

UniFFI 为每个启用 `uniffi` feature 的 crate **独立生成**一套绑定文件。虽然有 6 个 crate 导出，但 UniFFI 通过**外部类型 (external types)** 机制实现类型共享：
- 主模块 `matrix_sdk_ffi` 引用 `matrix_sdk`、`matrix_sdk_crypto` 等子模块中定义的类型
- 子模块的 Swift 文件定义纯数据类型，主模块通过 `import` 引用

### 5.2 matrix_sdk_ffi.swift 为何达 49K 行？

UniFFI 的代码生成非常**详尽**，对每个导出的类型/方法都生成完整的序列化/反序列化代码：

1. **FfiConverter 体系**（约 30% 行数）：为每个类型生成 `lift`/`lower`/`read`/`write` 方法
2. **协议 + 类双重定义**（约 20%）：每个 interface 生成 `Protocol` + `open class` 两套
3. **异步基础设施**（约 15%）：Future poll/complete/free 包装
4. **校验和验证**（约 10%）：599 个 checksum 调用 + 初始化函数
5. **外部类型桥接**（约 10%）：跨模块类型引用
6. **回调接口**（约 10%）：792 个 foreign trait 实现
7. **Doc 注释**（约 5%）：从 Rust 源码保留的文档注释

### 5.3 为什么每个 crate 生成 3 个文件 (.swift + .h + .modulemap)？

| 文件 | 用途 | 必须性 |
|------|------|--------|
| `.swift` | Swift API 实现 | 必须（唯一用户可见 API） |
| `.h` | C ABI 函数声明 | 必须（Swift 需要通过 C 桥接调用 Rust） |
| `.modulemap` | Clang 模块定义 | 必须（Xcode/SwiftPM 需要它来导入 C 头文件作为 Swift 模块） |
